# Event Attendance (RSVP) Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add an `EventAttendance` linkage between `Profile` and `EventContent` so users can RSVP `GOING` or `MAYBE` to events. Cancellation is hard-delete. Event creators are auto-attended as `GOING` in the same transaction as event creation.

**Architecture:** A new `EventAttendance` table with a composite PK `(eventContentId, profileId)` and a `status` enum (`GOING | MAYBE`). Hard-delete on cancel. RSVP state changes go through `EventService` via `prisma.eventAttendance.upsert`; cancels go through `deleteMany` (idempotent). The existing `EventService.create()` `$transaction` gains a sibling insert so every event has at least one attendee on creation. Routes mount under the existing `apps/backend/src/api/routes/content/event.route.ts`. Tests live alongside existing `event.service.spec.ts`.

**Tech Stack:** Prisma + Postgres, Fastify routes, Zod DTOs in `packages/shared/zod/event/`, Vitest with a hand-built Prisma mock factory in `apps/backend/src/test-utils/prisma.ts`. Frontend additions are out of scope for this plan — that will be a follow-up branch.

```ts
  fastify.post(
    '/:id/rsvp',
    {
      onRequest: [fastify.authenticate],
      config: rateLimitConfig(fastify, '1 minute', 30),
    },
    async (req, reply) => {
      const { id } = EventParamsSchema.parse(req.params)
      const profileId = req.session.profileId
      if (!profileId) return sendError(reply, 401, 'Profile required')
      const data = validateBody<RsvpPayload>(RsvpPayloadSchema, req, reply)
      if (!data) return
      try {
        await svc.rsvp(profileId, id, data.status)
        return reply.code(200).send({ success: true })
      } catch (err) {
        fastify.log.error(err)
        return sendError(reply, 500, 'Failed to RSVP')
      }
    }
  )

  fastify.delete(
    '/:id/rsvp',
    {
      onRequest: [fastify.authenticate],
      config: rateLimitConfig(fastify, '1 minute', 30),
    },
    async (req, reply) => {
      const { id } = EventParamsSchema.parse(req.params)
      const profileId = req.session.profileId
      if (!profileId) return sendError(reply, 401, 'Profile required')
      try {
        await svc.cancelRsvp(profileId, id)
        return reply.code(200).send({ success: true })
      } catch (err) {
        fastify.log.error(err)
        return sendError(reply, 500, 'Failed to cancel RSVP')
      }
    }
  )

  fastify.get(
    '/:id/attendees',
    { onRequest: [fastify.authenticate] },
    async (req, reply) => {
      const { id } = EventParamsSchema.parse(req.params)
      const { status } = AttendeeListQuerySchema.parse(req.query)
      try {
        const rows = await svc.listAttendees(id, status)
        const attendees = rows.map((r: any) => ({
          profile: r.profile,
          status: r.status,
          rsvpedAt: r.rsvpedAt,
        }))
        return reply.code(200).send({ success: true, attendees })
      } catch (err) {
        fastify.log.error(err)
        return sendError(reply, 500, 'Failed to list attendees')
      }
    }
  )
```

## Self-review notes (for the executing agent)

- The plan deliberately does NOT touch the frontend. RSVP UI is a follow-up.
- The plan does NOT add a `DECLINED` status. With hard-delete, "declined" and "absent" collapse into the same state (no row). If product feedback later requires distinguishing them, add it as a new minor.
- The plan does NOT add a "last attendee" or "creator cannot cancel their own RSVP" invariant. If the poster cancels, the event simply renders "0 going" — confirmed as acceptable in design discussion.
- Soft-deleted events keep their attendance rows. This is intentional; revisit only if it becomes a real problem.
- The route-test step (8.2) is conditional on the project having an existing route-spec pattern. If none exists, the service tests + manual smoke are the test floor — don't invent a new harness in this branch.
